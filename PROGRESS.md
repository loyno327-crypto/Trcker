## 1. Оригинальное ТЗ (без изменений)

Создай полноценный Telegram-бот с WebApp «Трекер привычек и распорядок дня».

### СТЕК ТЕХНОЛОГИЙ

Использовать:
- Python 3.12
- aiogram 3.x
- FastAPI
- SQLAlchemy ORM
- SQLite (архитектура должна позволять легко перейти на PostgreSQL)
- HTML
- CSS
- JavaScript
- Telegram WebApp SDK

Код должен быть чистым, модульным и легко расширяемым.

Использовать:
- type hints
- логирование
- обработку ошибок
- конфигурацию через .env
- разделение проекта на сервисы
- понятную структуру файлов
- комментарии только там, где они действительно нужны

### СТРУКТУРА ПРОЕКТА

  database.py

### ПРАВИЛО РАЗРАБОТКИ

Работать строго поэтапно. Каждый этап должен быть полностью рабочим.
Нельзя переходить к следующему этапу, пока предыдущий полностью не завершен.
После каждого этапа объяснять, что было сделано, и ждать подтверждения.
Нельзя писать весь проект сразу.

### WEBAPP

Создать современный WebApp. Интерфейс должен максимально повторять стиль
Telegram iOS и Telegram Android. Использовать: светлую тему, фирменные цвета
Telegram, современные иконки, плавные анимации, закругленные карточки, мягкие
тени, одинаковые отступы, минималистичный дизайн. Интерфейс должен выглядеть
как встроенная часть Telegram, а не как обычный сайт. Полностью адаптировать
под любые смартфоны. Все элементы должны работать быстро и плавно.

### НИЖНЕЕ МЕНЮ

Внизу WebApp сделать фиксированное меню. Кнопки: 📅 Календарь, ➕ Добавить,
📊 История. Меню всегда остается на экране.

### ГЛАВНАЯ СТРАНИЦА

Главная страница содержит календарь. Требования: отображение месяца,
переключение месяцев, каждая дата кликабельна, при выборе даты открывается
список задач. На календаре отображать состояние каждого дня: ⚪ Нет задач,
🟡 Есть задачи, 🟢 Все задачи выполнены, 🔴 Есть просроченные задачи.

### СПИСОК ЗАДАЧ

Каждая задача должна занимать ровно одну строку. Например:
`09:00 | Магазин | ✅ ❌ ✏️ 🗑 📅` или `09:00 Магазин ✅ ❌ ✏️ 🗑 📅`.
Запрещается переносить название/время/кнопки на новую строку. Если название
длинное — сокращается многоточием. Все кнопки всегда справа. Высота всех
строк одинаковая. Использовать Flexbox или Grid. Интерфейс максимально
компактный.

### КНОПКИ ЗАДАЧИ

Возле каждой задачи: ✅ Выполнено, ❌ Не выполнено, ✏️ Редактировать,
🗑 Удалить, 📅 Перенести. Все кнопки в одной строке.

### СОЗДАНИЕ ЗАДАЧИ

Поля: Название, Дата, Время, Повторение. Варианты повторения: один раз,
каждый день, по будням, по выходным, каждую неделю, каждый месяц.
Архитектура должна позволять позже добавить любые пользовательские
повторения.

### СОРТИРОВКА

После создания новой задачи список автоматически сортируется. Сначала по
времени. Если время одинаковое — по времени создания. Никакая сортировка
вручную пользователю не требуется.

### РЕДАКТИРОВАНИЕ

Можно изменить: название, дату, время, повторение. После сохранения задача
автоматически занимает правильное место в отсортированном списке.

### ПЕРЕНОС

При переносе пользователь выбирает новую дату и новое время. После сохранения
задача автоматически сортируется.

### ПОВТОРЯЮЩИЕСЯ ЗАДАЧИ

При изменении повторяющейся задачи спросить: «Изменить только эту задачу»
или «Изменить всю серию». То же самое при удалении.

### УВЕДОМЛЕНИЯ

Scheduler каждую минуту проверяет задачи. Когда наступает время задачи, бот
отправляет сообщение, например:
```
🔔 Напоминание
Была задача:
🛒 Сходить в магазин
🕛 12:00
```
Кнопки: ✅ Выполнено, ❌ Не сделал, 📅 Перенести. После нажатия сразу
обновляется статус задачи.

### ИСТОРИЯ

Отдельная страница «История», открывается кнопкой 📊. Интерфейс очень
минималистичный. Отображать периоды: Сегодня, Неделя, Месяц, Все время.
Показатели: ✅ Выполнено задач, ❌ Не выполнено задач, 📈 Процент выполнения,
🔥 Текущая серия полностью выполненных дней, 🏆 Лучшая серия, 📅 Самый
продуктивный день, последние выполненные задачи, последние пропущенные
задачи. Добавить небольшой прогресс-бар или круговой индикатор. Не
использовать тяжелые графики.

### ДОСТИЖЕНИЯ

В разделе «История» блок «Достижения», минималистично, карточками с иконкой.
Например: 🎉 Первая выполненная задача, 🔥 7 дней подряд, 🏆 30 дней подряд,
💯 Выполнено 100 задач, ⭐ Выполнено 500 задач, 📅 Неделя без пропусков,
🌟 Месяц без пропусков. Архитектура должна позволять легко добавлять новые
достижения.

### БАЗА ДАННЫХ

SQLAlchemy ORM. Модели: users, tasks, task_series, task_history,
achievements. Все связи реализованы правильно.

### ТРЕБОВАНИЯ К КОДУ

Современные практики. Не дублировать код. Логику разделять по сервисам.
Функции небольшие и понятные. Код готов к масштабированию.

### ВАЖНО

Не писать весь проект сразу. Работать только по этапам. Каждый этап должен
быть полностью закончен. После окончания этапа подробно объяснять, что было
сделано. Только после подтверждения переходить к следующему этапу.

## 3. Прогресс по этапам

### Этап 1 — Каркас проекта, БД, `/start` — ✅ ЗАВЕРШЁН

- Структура проекта создана (`app/bot.py`, `config.py`, `database.py`,
  `models.py`, `scheduler.py`, `handlers/`, `services/`, `web/`).
- Конфигурация через `.env` (pydantic-settings).
- Логирование через `logging`, единая настройка в `app/logger.py`.
- Async SQLAlchemy: `engine`, `session_scope()`, `init_db()`.
- ORM-модели: `User`, `TaskSeries`, `Task`, `TaskHistory`, `Achievement`,
  `UserAchievement`.
- Рабочий бот на aiogram 3.x, команда `/start` регистрирует/обновляет
  пользователя в БД.

### Этап 2 — Сервисный слой для задач — ✅ ЗАВЕРШЁН

- `app/services/recurrence.py` — чистая функция `dates_for_series()`,
  вычисляет даты вхождения серии в диапазон для ONCE/DAILY/WEEKDAYS/
  WEEKENDS/WEEKLY/MONTHLY (MONTHLY корректно пропускает месяцы без нужного
  числа, напр. 31 января → пропуск февраля). CUSTOM зарезервирован.
- `app/models.py`: в `TaskSeries` добавлено поле `last_generated_until` —
  генерация экземпляров всегда идёт только вперёд от этой отметки, поэтому
  перенос одного экземпляра не приводит к повторному появлению задачи на
  старой дате.
- `app/config.py`: добавлен `task_generation_horizon_days` (по умолчанию 60) —
  на сколько дней вперёд заранее генерируются экземпляры серии.
- `app/services/exceptions.py` — `TaskNotFoundError`, `TaskNotOwnedError`.
- `app/services/task_service.py`:
  - `create_task()` — создаёт `TaskSeries` (в т.ч. для разовой задачи) и
    генерирует экземпляры `Task` на горизонт вперёд;
  - `list_tasks_for_date()` — задачи на дату, сортировка по времени, затем
    по времени создания (как того требует ТЗ, вручную сортировать не нужно);
  - `update_task(scope="this"|"series")` — редактирование одного экземпляра
    или всей серии (у серии дата не меняется — для этого `move_task`);
  - `move_task()` — перенос всегда влияет только на конкретный экземпляр;
  - `delete_task(scope="this"|"series")` — удаление экземпляра или всей
    серии целиком (каскадно);
  - `set_task_status()` — меняет статус и пишет запись в `TaskHistory`.
- Логика повторений проверена изолированным тестовым прогоном (DAILY,
  WEEKDAYS, WEEKENDS, WEEKLY, MONTHLY, ONCE) — результаты корректны.

> Полный прогон `task_service.py` с реальной БД пока не выполнялся в этой
> среде из-за отсутствия сетевого доступа для установки зависимостей
> (aiogram/SQLAlchemy/aiosqlite). Нужно проверить локально командой из
> README после `pip install -r requirements.txt`.

### Этап 3 — FastAPI + REST API для WebApp — ✅ ЗАВЕРШЁН

- `app/web/telegram_auth.py` — проверка подписи Telegram WebApp `initData`
  по официальному алгоритму (HMAC-SHA256, secret_key из bot_token),
  плюс проверка срока действия (`auth_date`, не старше 24 часов).
  Алгоритм подписи/проверки прогнан отдельным изолированным тестом
  (сборка initData → валидация → подмена данных корректно отклоняется).
- `app/web/deps.py` — FastAPI-зависимости: `get_db_session` (сессия БД,
  коммитится на успешный ответ — важно, т.к. GET-эндпоинты как побочный
  эффект дозаписывают сгенерированные задачи и могут создать пользователя
  при первом обращении) и `get_current_user` (читает заголовок
  `Authorization: tma <initData>`, возвращает `User` из БД).
- `app/web/schemas.py` — Pydantic-схемы ответов: `CalendarOut`, `TaskOut`,
  `TaskListOut`.
- `app/web/routers/calendar.py` — `GET /api/calendar?year=&month=`,
  состояние каждого дня месяца (⚪/🟡/🟢/🔴).
- `app/web/routers/tasks.py` — `GET /api/tasks?date=YYYY-MM-DD`, список
  задач на дату (уже отсортирован сервисным слоем).
- `app/web/main.py` — сборка FastAPI-приложения, CORS, `/health`,
  роутеры примонтированы под `/api`.
- `app/database.py`: `get_session()` теперь коммитит при успехе и
  откатывает при ошибке (раньше не коммитил — GET-запросы теряли
  побочные записи).
- `app/services/task_service.py`: добавлены `list_tasks_for_period()`,
  `get_month_overview()` и `_ensure_generated_for_range()` — при запросе
  месяца/даты дальше исходного горизонта генерации (`task_generation_horizon_days`)
  задачи серии автогенерируются "по требованию", не только заранее.
- `scripts/make_test_init_data.py` — генерирует валидно подписанный
  `initData` для ручного теста API через curl/Swagger (`/docs`) без
  реального Telegram-клиента.

**Как проверить (после `pip install -r requirements.txt`, `.env` с
`BOT_TOKEN`):**

```bash
uvicorn app.web.main:app --reload --port 8000
python -m scripts.make_test_init_data          # выведет строку Authorization
```

Открыть `http://localhost:8000/docs`, в Authorize вставить `tma <initData>`
из вывода скрипта (без слова Authorization и заголовка), выполнить
`GET /api/calendar?year=2026&month=7` и `GET /api/tasks?date=2026-07-23`.
Либо через curl с заголовком `-H "Authorization: tma <initData>"`.

> Полный прогон с реальной БД пока не выполнялся в этой среде из-за
> отсутствия сетевого доступа для установки зависимостей — код проверен
> `py_compile` (весь проект компилируется) и изолированным тестом
> HMAC-алгоритма initData (сборка/проверка/детект подделки — пройдено).
> Нужно прогнать локально командой выше.

### Этап 5 — Frontend WebApp: создание/редактирование/перенос задачи — ✅ ЗАВЕРШЁН

- `app/web/schemas.py` — добавлены схемы запросов: `TaskCreateIn` (title,
  date, time, repeat_type), `TaskUpdateIn` (scope="this"|"series" + поля),
  `TaskMoveIn` (date, time), `TaskStatusIn` (status), и `TaskMutationOut` —
  единый формат ответа после любой мутации (полная актуальная задача, чтобы
  фронтенду не нужно было перезапрашивать список ради одной строки).
  `TaskOut` дополнен полем `repeat_type` — нужно фронтенду, чтобы понимать,
  какой текст повторения показывать в диалоге "эта задача / вся серия".
- `app/web/routers/tasks.py` — новые эндпоинты поверх уже готового
  `task_service` (бизнес-логика не менялась, только API-обвязка):
  - `POST /api/tasks` — создание (разовой задачи или серии);
  - `PATCH /api/tasks/{id}` — редактирование, `scope=this` меняет
    заголовок/дату/время только экземпляра, `scope=series` — заголовок/время/
    повторение всей серии (дата серии этим эндпоинтом не меняется — для этого
    перенос);
  - `POST /api/tasks/{id}/move` — перенос (всегда только один экземпляр, как
    того требует ТЗ);
  - `POST /api/tasks/{id}/status` — отметка выполнено/не выполнено;
  - `DELETE /api/tasks/{id}?scope=this|series` — удаление экземпляра или
    всей серии.
  `TaskNotFoundError`/`TaskNotOwnedError` из сервисного слоя превращаются в
  404/403, некорректный `scope` — в 400.
- `app/services/task_service.py`: `get_owned_task()` и
  `list_tasks_for_date()` теперь используют `selectinload(Task.series)` —
  чтобы отдавать `repeat_type` в API без лишних неявных ленивых подгрузок в
  асинхронном контексте (сама бизнес-логика не менялась).
- Фронтенд (`index.html`/`style.css`/`app.js`):
  - экран **➕ Добавить** — рабочая форма (название, дата, время, повторение),
    после создания переключает на календарь и сразу открывает список задач
    на дату созданной задачи;

### Инфраструктура — адаптация запуска под хостинг BotHost

Пользователь столкнулся с ошибкой запуска на BotHost: платформа пыталась

- Добавлен `main.py` в корне проекта — единая точка входа, которая в одном

  `uvicorn app.web.main:app --reload` отдельно) продолжают работать как
  раньше — `main.py` это дополнение специально для продакшн-хостинга, а не

### Этап 8 — Достижения — ✅ ЗАВЕРШЁН

- `app/services/achievement_service.py` — новый сервис, единственное место с
  правилами достижений:
  - `ACHIEVEMENT_CATALOG` — список из 7 достижений по примеру из ТЗ (код,
    заголовок, описание, иконка): `first_done` 🎉, `streak_7` 🔥, `streak_30`
    🏆, `done_100` 💯, `done_500` ⭐, `week_clean` 📅, `month_clean` 🌟.
    Порядок в этом списке — он же порядок отображения карточек. Чтобы
    добавить новое достижение в будущем (требование ТЗ "архитектура должна
    позволять легко добавлять новые достижения"), нужно дописать сюда одну
    запись и один checker в `CHECKERS` — сидинг в БД и выдача пользователям
    подхватят его без остальных изменений в коде;
  - **🔥/🏆 "N дней подряд"** — используют ту же серию, что и статистика на
    странице «История» (`stats_service.compute_streaks`, `best_streak`, а не
    `current_streak` — достижение не должно "сгорать", если серия позже
    прервалась). Функция `_compute_streaks` в `stats_service.py` для этого
    сделана публичной (`compute_streaks`), чтобы не дублировать алгоритм
    серии во второй раз;
  - **📅/🌟 "без пропусков за неделю/месяц"** — осознанно отдельная, более
    узкая метрика (`_clean_window`): смотрит только на последние 7/30
    завершившихся календарных дней (сегодня не в счёт, оно может быть ещё не
    закончено) и требует отсутствия просроченных дней среди дней-с-задачами
    в этом окне; в отличие от серии, не требует, чтобы каждый день окна
    содержал задачи, и не смотрит на историю до этого окна. Из-за этого это
    не дубликат 🔥/🏆, а самостоятельное достижение: можно получить "неделю
    без пропусков" при серии короче 7 (из-за "пустых" дней внутри недели), и
    наоборот. Требует хотя бы одного дня с задачами в окне — иначе новый
    пользователь без единой задачи получал бы достижение сразу;
  - `check_and_award(session, user)` — сравнивает "снимок" статистики
    пользователя (задачи за всё время, лучшая серия, чистота окон 7/30 дней)
    со всеми `CHECKERS`, разблокирует новые (создаёт `UserAchievement`) и
    возвращает только что разблокированные — чтобы вызывающая сторона могла
    показать уведомление. Уже разблокированные достижения повторно не
    выдаются и не отбираются обратно, даже если показатель впоследствии
    "ухудшился" (например, пользователь снял отметку с задачи) — это
    исторический факт, как и принято в трекерах привычек;
  - `list_achievements(session, user)` — весь каталог с датой разблокировки
    у пользователя (`None`, если не разблокировано) — для страницы «История».

  перезаписывает их поля), вызывается из `init_db()` после создания таблиц.
  Идемпотентно на каждом старте; новое достижение, дописанное в каталог
  позже, появится в БД при следующем запуске без миграций и без потери уже

- `app/web/routers/tasks.py` — эндпоинт `POST /tasks/{id}/status` после
  `task_service.set_task_status` вызывает `achievement_service.check_and_award`
  и возвращает разблокированные достижения в ответе (только этот эндпоинт
  может на них повлиять — остальные мутации задач на выполненные/просроченные
  счётчики не влияют).
- `app/handlers/reminders.py` — кнопки ✅/❌ под напоминанием тоже вызывают
  `check_and_award`; на каждое новое достижение бот дополнительным сообщением
  в чат пишет «{иконка} Новое достижение: «{название}»».
- Фронтенд:
  - `index.html` — на экране «История» под период-зависимой статистикой
    добавлен отдельный (не зависящий от вкладки периода) блок «🏅 Достижения»
    с сеткой карточек;
  - `app.js` — `loadAchievements()`/`renderAchievements()`: карточки с
    иконкой и названием, заблокированные — приглушены и обесцвечены
    (`filter: grayscale`), подсказка (`title`) с описанием. Грузится при
    каждом открытии вкладки «История» (вместе с `loadHistory`), не привязана
    к переключению периодов. При отметке задачи ✅/❌ во всплывающем списке
    задач ответ API может содержать `new_achievements` —
    `announceNewAchievements()` показывает тост «Новое достижение: …» и, если
    экран «История» уже открыт, сразу обновляет сетку, чтобы не показывать
    устаревшую (ещё заблокированную) карточку;
  - `style.css` — `.achievements-block`/`.achievements-grid`/
    `.achievement-card` (сетка 3 колонки, переиспользует тени и радиусы
    карточек статистики), `.locked` для невыполненных.

**Как проверить (после `pip install -r requirements.txt`, `.env` с
`BOT_TOKEN` и `WEBAPP_URL`):**

```bash
uvicorn app.web.main:app --reload --port 8000
python -m scripts.make_test_init_data
```

Открыть `/?init_data=...`, перейти на вкладку «История» — под статистикой
должен появиться блок «Достижения» с 7 приглушёнными карточками. Выполнить
первую задачу (✅) — должен появиться тост «Новое достижение: 🎉 Первая
выполненная задача», а карточка в сетке — стать активной (цветной). Чтобы
проверить серию, создать и выполнить задачи 7 дней подряд (или перенести уже
выполненные на подряд идущие прошлые даты через `PATCH /api/tasks/{id}` с
`scope=this` и `date` в прошлом через `/docs`) — должно разблокироваться
🔥 «7 дней подряд». Также можно напрямую дёрнуть `GET /api/achievements`
через `/docs`, чтобы увидеть `unlocked_at` каждого достижения.
